# LightBridge 模块开发与发布指南（Phase 2：静态 index.json + GitHub Releases）

本文说明如何在 `WilliamWang1721/LightBridge` 仓库里创建一个新模块，并将其发布到远程 Marketplace（Core 默认会从远程静态索引加载模块列表）。

## 0. 背景与约定

### 0.1 Marketplace 目录约定

- 远程 Marketplace 目录固定为：`market/MODULES/`
- 静态索引固定为：`market/MODULES/index.json`

说明：

- 不使用仓库根目录 `MODULES/`：在 macOS 常见的“大小写不敏感”文件系统下，根目录已存在 `modules/` 源码目录，无法再创建同级 `MODULES/`。

### 0.2 Core 默认模块源（Phase 2）

Core 默认读取的 Marketplace 索引为：

```text
https://raw.githubusercontent.com/WilliamWang1721/LightBridge/main/market/MODULES/index.json
```

你仍然可以通过环境变量覆盖（比如切回 Phase 1 的 GitHub 目录扫描）：

```bash
export LIGHTBRIDGE_MODULE_INDEX=github:WilliamWang1721/LightBridge/market/MODULES@main
```

## 1. 创建一个模块（源码结构）

推荐在仓库 `modules/` 下创建一个新目录：

```text
modules/<module_id>/
  manifest.json
  README.md                 # 可选，但强烈推荐
  package.sh                # 打包脚本：产出 universal zip
  cmd/<module_binary>/
    main.go                 # 模块入口
```

说明：

- `<module_id>` 是模块唯一 ID（建议全小写 + `-`）。
- `<module_binary>` 通常与 `<module_id>` 相同，便于一致性。
- `package.sh` 的目标产物是一个“单 zip 覆盖多平台”的 Universal ZIP（见第 3 节）。

## 2. 编写 manifest.json（必需）

Core 安装模块时会校验 `manifest.json`。安装完成后，Core 会按 `entrypoints` 在本机启动模块进程，并做健康检查。

### 2.1 最关键字段

manifest 必须包含：

- `id`：模块 ID（与目录名建议一致）
- `version`：模块版本号（建议 SemVer，例如 `0.1.0`）
- `entrypoints`：平台启动命令映射（建议覆盖 4 个 key）
- `services[]`：服务声明（当前 Core 仅支持 `kind: "provider"`）

支持的 provider 协议（`services[].protocol`）：

- `http_openai`
- `http_rpc`
- `grpc_chat`
- `codex`

### 2.2 Universal ZIP 的 entrypoints（推荐固定写法）

由于 Marketplace 的 `ModuleEntry` 只有一个 `download_url`，Core 不会按平台选择不同 zip，因此推荐用“一个 zip 覆盖多平台”的方式发布，并在 `entrypoints` 里写死 4 个 OS/ARCH：

```json
{
  "entrypoints": {
    "darwin/arm64": { "command": "bin/darwin/arm64/<module-binary>", "args": [] },
    "darwin/amd64": { "command": "bin/darwin/amd64/<module-binary>", "args": [] },
    "linux/arm64":  { "command": "bin/linux/arm64/<module-binary>",  "args": [] },
    "linux/amd64":  { "command": "bin/linux/amd64/<module-binary>",  "args": [] },
    "default":      { "command": "bin/darwin/arm64/<module-binary>", "args": [] }
  }
}
```

注意：

- `command` 必须是相对路径，且相对 `manifest.json` 所在目录可找到该文件。
- `default` 可保留兜底，但不要只写 `default`（否则会在非匹配平台直接失败）。

### 2.3 services/health（强烈建议）

建议所有 HTTP 类协议（`http_openai` / `http_rpc` / `codex`）都提供一个 HTTP 健康检查：

```json
{
  "services": [
    {
      "kind": "provider",
      "protocol": "http_rpc",
      "health": { "type": "http", "path": "/health" },
      "expose_provider_aliases": []
    }
  ]
}
```

Core 启动模块后会在 10 秒内反复探测 `GET http://127.0.0.1:<LIGHTBRIDGE_HTTP_PORT>/health`，返回 `200` 才会判定健康。

## 3. 实现模块（运行时约定）

Core 启动模块子进程时会注入以下环境变量（节选）：

- `LIGHTBRIDGE_MODULE_ID`
- `LIGHTBRIDGE_DATA_DIR`（模块私有数据目录：`<DATA_DIR>/module_data/<module_id>`）
- `LIGHTBRIDGE_CONFIG_PATH`（模块配置文件：`<DATA_DIR>/module_data/<module_id>/config.json`）
- `LIGHTBRIDGE_HTTP_PORT`
- `LIGHTBRIDGE_GRPC_PORT`（仅当你声明了 gRPC 协议时会使用）

实现侧约定：

- 仅监听回环地址：`127.0.0.1:<port>`
- 在 `GET /health` 返回 `200`（建议尽量快；Core 超时时间为 10 秒）
- 从 `LIGHTBRIDGE_CONFIG_PATH` 读取 JSON 配置；写私有数据到 `LIGHTBRIDGE_DATA_DIR`

## 4. 打包模块（Universal ZIP）

### 4.1 ZIP 内容结构（解压后即安装目录）

推荐 Universal ZIP 解压后的目录结构如下（`manifest.json` 放在根目录最简单）：

```text
manifest.json
README.md                 # 可选
bin/darwin/arm64/<module-binary>
bin/darwin/amd64/<module-binary>
bin/linux/arm64/<module-binary>
bin/linux/amd64/<module-binary>
```

### 4.2 产物命名

建议 zip 资产命名为：

```text
<module_id>_<version>_universal.zip
```

仓库现有模块的 `package.sh` 都遵循该命名，可以直接参考：

- `modules/openai-codex-oauth/package.sh`
- `modules/passkey-login/package.sh`
- `modules/totp-2fa-login/package.sh`

### 4.3 本地打包

```bash
bash modules/<module_id>/package.sh
ls -la modules/<module_id>/dist/
```

你会得到：

- `modules/<module_id>/dist/<module_id>_<version>_universal.zip`
- 以及编译出的 4 平台二进制位于 `modules/<module_id>/dist/bin/...`

计算 sha256（用于 Marketplace 强校验）：

```bash
shasum -a 256 "modules/<module_id>/dist/<module_id>_<version>_universal.zip"
```

## 5. 发布到远程 Marketplace（Phase 2）

Phase 2 的原则：

- zip 不提交进 git
- zip 上传到 GitHub Releases
- `market/MODULES/index.json` 只维护“每个 id 一条最新版本”，并包含 `download_url + sha256`

### 5.1（可选）自动发布：GitHub Actions

仓库提供了 workflow：`.github/workflows/publish-module.yml`

触发方式：

- push tag：`module-<module_id>-v<version>`
- 或 workflow_dispatch（Actions 页面手动触发）

workflow 会：

1. 运行 `modules/<module_id>/package.sh` 生成 universal zip
2. 创建/更新 release 并上传 zip asset
3. 更新并提交 `market/MODULES/index.json`

如果你发现 Actions 无法启动（例如账单/权限导致 job 被拒绝），请走 5.2 手动发布流程。

### 5.2 手动发布（推荐兜底流程）

以下示例以模块 `<module_id>`、版本 `<version>` 为例。

#### Step 1：打包并计算 sha256

```bash
module_id="<module_id>"
version="<version>"

bash "modules/${module_id}/package.sh"
zip_path="modules/${module_id}/dist/${module_id}_${version}_universal.zip"
sha="$(shasum -a 256 "${zip_path}" | awk '{print $1}')"
echo "${sha}"
```

#### Step 2：创建 tag 并 push

tag 命名约定：

```text
module-<module_id>-v<version>
```

```bash
tag="module-${module_id}-v${version}"
git tag "${tag}"
git push origin "${tag}"
```

#### Step 3：创建 GitHub Release 并上传 asset

需要 `gh` 已登录（`gh auth status`）。

```bash
gh release create "${tag}" "${zip_path}" \
  --title "Module ${module_id} v${version}" \
  --notes "Universal module zip (darwin/linux, arm64/amd64)." \
  --target main
```

asset 对应的下载链接格式为：

```text
https://github.com/WilliamWang1721/LightBridge/releases/download/<tag>/<asset-file>
```

例如：

```text
https://github.com/WilliamWang1721/LightBridge/releases/download/module-<module_id>-v<version>/<module_id>_<version>_universal.zip
```

#### Step 4：更新静态索引 market/MODULES/index.json

仓库提供了脚本：`scripts/update_modules_index.py`（会按 `id` 覆盖旧条目，保持每个 `id` 唯一，并更新 `generated_at`）。

```bash
download_url="https://github.com/WilliamWang1721/LightBridge/releases/download/${tag}/${module_id}_${version}_universal.zip"
homepage="https://github.com/WilliamWang1721/LightBridge/tree/main/modules/${module_id}"

python3 scripts/update_modules_index.py \
  --index "market/MODULES/index.json" \
  --manifest "modules/${module_id}/manifest.json" \
  --download-url "${download_url}" \
  --sha256 "${sha}" \
  --homepage "${homepage}"
```

提交并 push（只需要提交 index.json）：

```bash
git add market/MODULES/index.json
git commit -m "MODULES: update index for ${module_id} ${version}"
git push origin main
```

## 6. 发布后验收（建议检查清单）

1. Release 资产可下载（HTTP 302 到 `release-assets.githubusercontent.com` 是正常的）：

```bash
curl -I "https://github.com/WilliamWang1721/LightBridge/releases/download/${tag}/${module_id}_${version}_universal.zip"
```

2. `market/MODULES/index.json` 已包含该模块的条目，且 `sha256` 与本地计算一致。
3. 启动 Core 后打开：

```text
http://127.0.0.1:3210/admin/marketplace
```

确认模块列表可见、安装成功、启用后能在 10 秒内通过健康检查。

## 7. 常见坑位

- `manifest.version` 与 tag 版本不一致：Marketplace 会以 `manifest.version` 为准，建议永远保持一致。
- `sha256` 不一致：Core 安装时会强校验 sha256，填错会直接安装失败。
- entrypoints 写错平台 key：key 必须匹配 `darwin/arm64`、`darwin/amd64`、`linux/arm64`、`linux/amd64`（大小写敏感）。
- 健康检查不通过：确认监听的是 `127.0.0.1:${LIGHTBRIDGE_HTTP_PORT}`，且 `GET /health` 返回 200。
- 无法访问 `raw.githubusercontent.com`：可临时切回 `github:` 目录扫描源，或把静态 index 复制到一个你可访问的静态站点再改 `LIGHTBRIDGE_MODULE_INDEX`。
