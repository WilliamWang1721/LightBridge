根据这个计划进行开发：# LightBridge（Go + 模块市场 + 多协议供应源）实现计划

## 0. 目标与边界

### 目标（你要的核心价值）
- **对下游**：提供一个统一的 **OpenAI 兼容 Base URL**（尽量覆盖 `/v1/*` 全家桶），让 Cherry Studio / Cline / 自建应用能直接用。
- **对上游（供应源）**：通过模块化 Provider，把 **OpenAI / Anthropic / Gemini** 等不同协议的供应源统一接入，并支持同名模型的**多供应源细粒度选择**（`model@provider` 变体 + 可视化映射 + 默认路由）。
- **极强模块化**：用户只安装“基础框架（MVP）”就能跑起来；需要更多能力（供应源、远程、分发、用户端）再从**内置模块市场**安装。

### 明确不做/不承诺（避免项目方向跑偏）
- **不把“额度 A 变成额度 B”当成核心承诺**：LightBridge 是统一网关/协议转换与路由框架；任何供应源模块必须自行合法合规可用（你可从开源项目抽模块，但是否符合各平台条款需你自行评估）。
- MVP 不内置“高级分发（批量 Key/限额/额度卡/用户门户）”，这些作为后续模块交付。

## 1. 技术选型（已决策）

- **核心语言**：Go（跨平台 macOS/Windows/Linux，避免 CGO，发布单文件）
- **存储**：SQLite（纯 Go 驱动：`modernc.org/sqlite`），数据目录跨平台标准路径
- **模块通信协议**：**同时支持**
  - **HTTP+JSON / HTTP(OpenAI upstream)**：最大兼容既有开源代理项目（直接跑起来就能用）
  - **gRPC**：给新模块/高性能流式做标准化接口
- **模块运行方式**：本地子进程（由核心拉起、分配端口、做健康检查）
- **模块市场**：官方精选（GitHub 索引 + ZIP 包 + SHA256 校验）
- **日志**：默认仅元数据（不落盘提示词/响应正文）
- **密钥存储**：明文存 SQLite（便于维护迁移；在文档里强提示风险与文件权限要求）
- **开源协议**：核心 GPLv3（便于直接合并/改造 GPLv3 代码与模块）

## 2. 整体架构（Microkernel + Providers + Modules）

### 核心进程（LightBridge Core）职责
1. **南向（下游）HTTP 网关**：对外暴露 OpenAI 兼容 API（`/v1/*`），支持 SSE 流式
2. **路由与模型注册表**：
   - `model` → `provider`（可手动映射）
   - 同名模型多供应源 → 以 `model@providerAlias` 方式暴露“变体”
3. **Provider 管理**：
   - 内置 Provider：`forward`（OpenAI upstream 转发）、`anthropic`（Claude 直连转换）
   - 模块 Provider：HTTP(OpenAI upstream) / HTTP(RPC) / gRPC
4. **模块管理器**：安装/启用/禁用/更新/卸载模块，端口分配，健康检查，日志采集
5. **管理台（内置 Web UI）**：配置、Providers、Models、Marketplace、Logs、Docs（指导手册）
6. **基础 API Key 管理**：下游访问鉴权（单/多 key 都可做成 core 能力；“限额/号池/额度卡”留给分发模块）

### 模块类型（未来扩展点）
- **Provider 模块**：提供模型/生成能力（OpenAI/Claude/Gemini 等上游）
- **Runtime 模块（服务器能力）**：远程访问、反代、TLS、IP 白名单、Rate limit 等
- **Distribution 模块（高级分发）**：批量 Key、限额、专属号池、额度卡、用户门户
- **UI 模块**：给管理员/用户追加页面与 API

## 3. 下游（对外）OpenAI 兼容 API 设计

### 3.1 端口与路径（单端口承载）
- 同一端口同时提供：
  - `GET/POST /v1/*`：OpenAI 兼容 API（对下游客户端）
  - `GET /admin/*`：管理台页面（管理员）
  - `POST /admin/api/*`：管理台后台 API（管理员）
  - `GET /m/<moduleId>/*`：模块静态资源（可选）
  - `GET /healthz`：核心健康

### 3.2 “尽量全”如何实现（可落地方案）
- **所有 `/v1/*` 请求都先进入统一路由层**，由路由选择 Provider：
  - 如果 Provider 是 **forward / 模块(OpenAI upstream)**：**直接透传原始 HTTP 方法+路径+body+stream**
  - 如果 Provider 是 **anthropic / Gemini 等非 OpenAI 协议**：只对已实现的端点做协议转换（MVP 必做 `POST /v1/chat/completions`），其他端点返回 OpenAI 风格错误（如 `501_not_supported`），并在 UI/Docs 明确“该 Provider 能力矩阵”

> 这样做到：对下游来说“Base URL 是全的”，但不同 Provider 的可用端点不同；forward 类 Provider 能覆盖最多。

## 4. 细粒度路由规则（你要的“同名模型多供应源可选”）

### 4.1 核心机制：虚拟模型（Virtual Model）+ 变体（Variant）
- 对外暴露的模型 ID：
  - **基础名**：`claude-opus-4-5`（走默认映射）
  - **变体名**：`claude-opus-4-5@kiro`、`claude-opus-4-5@azure`（强制指定供应源）
- `/v1/models` 返回：
  1. 所有已配置的虚拟模型（基础名）
  2. 自动生成的“供应源变体模型”（`rawModel@providerAlias`），确保“同一个 Base URL 能列出所有当前支持的 Model”

### 4.2 路由算法（确定性、可控、可扩展）
1. 解析请求 `model`：
   - 若匹配 `(.+)@(.+)` → 直接锁定 `providerAlias`
   - 否则按虚拟模型映射表查默认 Provider（可多个候选）
2. 若一个虚拟模型有多个候选 Provider：
   - 先按 `priority`（小优先）
   - 再按 `weight` 做加权随机
   - 过滤掉不健康 Provider（健康检查失败/被禁用）
3. 若找不到映射：
   - `claude-*` 默认走 anthropic
   - 其它默认走 forward

### 4.3 管理台能力（MVP 必须有）
- Providers 页面：添加/编辑 Provider（forward、anthropic、模块 Provider）
- Models 页面：
  - 手动创建/编辑虚拟模型映射（基础名 → 一个或多个 Provider route）
  - 一键从 Provider 拉取 models 并生成 `model@provider` 变体（缓存刷新）

## 5. 内置 Providers（MVP 必交付）

### 5.1 Forward Provider（OpenAI upstream 转发）
- 配置项：
  - `base_url`（例如 `https://api.openai.com/v1` 或某个模块启动后的本地 URL）
  - `api_key`（写入 `Authorization: Bearer ...`）
  - `extra_headers`（可选）
  - `model_remap`（可选：虚拟模型 → 上游模型）
- 能力：
  - **透传全部 `/v1/*`**（含 SSE 流式），满足“尽量全”的大头需求

### 5.2 Anthropic Provider（Claude 直连）
- 配置项：
  - `anthropic_api_key`
  - `base_url`（默认官方）
  - `default_model_list`（用于 `/v1/models` 展示；也允许手动输入）
- MVP 实现端点：
  - `POST /v1/chat/completions`（含 stream）
- 转换要点（决策固定）
  - OpenAI messages → Anthropic messages
  - SSE chunk → OpenAI SSE（`data: {...}\n\n` + `[DONE]`）
  - 统一错误映射为 OpenAI error schema（`{error:{message,type,code}}`）
  - usage：优先用上游返回；无则估算（后续可模块化 tokenizer）

## 6. 模块系统与市场（MVP 必交付）

### 6.1 模块索引（GitHub）
- 默认索引 URL（可在设置里覆盖）：`https://.../lightbridge-modules/index.json`
- `index.json` 结构（固定字段，便于 UI 展示与校验）
  - `generated_at`
  - `min_core_version`
  - `modules[]`: `{id,name,version,description,license,tags,protocols,download_url,sha256,homepage}`

### 6.2 模块包格式（ZIP）
ZIP 内必须包含：
- `manifest.json`（见 6.3）
- `LICENSE`
- `README.md`
- `bin/` 或 `entry/`（各平台入口）

### 6.3 manifest.json（决定性规范）
必须字段（MVP 定稿）：
- `id`（slug，唯一）
- `name`
- `version`
- `license`
- `min_core_version`
- `entrypoints`（按 OS/ARCH 给出可执行入口与参数）
- `services[]`：模块提供的能力声明
  - Provider 服务：
    - `kind: "provider"`
    - `protocol: "http_openai" | "http_rpc" | "grpc_chat"`
    - `health`: `{type:"http", path:"/health"} | {type:"grpc"}`
    - `expose_provider_aliases[]`（模块启动后注册哪些 providerAlias）
- `config_schema`（JSON Schema，给管理台自动生成表单）
- `config_defaults`

### 6.4 模块运行时契约（Core → Module）
Core 启动模块时注入：
- `LIGHTBRIDGE_MODULE_ID`
- `LIGHTBRIDGE_DATA_DIR`（模块私有数据目录）
- `LIGHTBRIDGE_CONFIG_PATH`（模块配置 JSON 文件路径）
- `LIGHTBRIDGE_HTTP_PORT`（若该 module service 需要 HTTP）
- `LIGHTBRIDGE_GRPC_PORT`（若需要 gRPC）
- `LIGHTBRIDGE_LOG_LEVEL`

模块要求：
- 绑定 `127.0.0.1:<port>`
- 提供健康检查：
  - HTTP：`GET /health` 返回 200
  - gRPC：实现标准 health service

### 6.5 Core 对模块 Provider 的接入方式（MVP 必须都能跑）
- `http_openai`：Core 把选中 provider 的请求 **原样转发**到 `http://127.0.0.1:<port>/v1/*`
- `grpc_chat`：Core 仅把 `chat/completions` 转成 gRPC 调用（用于“非 OpenAI 协议的模块”）
- `http_rpc`：保留接口位（与 grpc_chat 同语义的 HTTP 版），用于兼容不想上 gRPC 的新模块

### 6.6 Marketplace UI（MVP 页面）
- 列表页：展示模块、版本、协议、License、安装状态
- 详情页：README 渲染、安装/卸载、启用/禁用、配置表单（JSON Schema 自动表单）
- 安装流程：下载 ZIP → 校验 SHA256 → 解压 → 写入 SQLite → 启用即拉起进程 → health ok → 注册 providerAliases 与 models

## 7. SQLite 数据模型（MVP 固化）

建议表（字段名固定，便于后续模块复用）：
- `settings`：`key TEXT PRIMARY KEY, value TEXT`
- `admin_users`：`id, username, password_hash, created_at`
- `client_api_keys`：`id, key, name, enabled, created_at, last_used_at`
- `providers`：
  - `id`（providerAlias）
  - `type`（builtin/module）
  - `protocol`（forward/anthropic/http_openai/grpc_chat/http_rpc）
  - `endpoint`（base_url 或 localhost:port）
  - `config_json`
  - `enabled`
  - `health_status`/`last_check_at`
- `models`（虚拟模型表）：
  - `id`（外显 model id）
  - `display_name`
  - `enabled`
- `model_routes`：
  - `model_id`
  - `provider_id`
  - `upstream_model`
  - `priority`
  - `weight`
  - `enabled`
- `modules_installed`：
  - `id`
  - `version`
  - `install_path`
  - `enabled`
  - `protocols`
  - `sha256`
  - `installed_at`
- `module_runtime`：
  - `module_id`
  - `pid`
  - `http_port`
  - `grpc_port`
  - `status`
  - `last_start_at`
- `request_logs_meta`（仅元数据，带保留策略）
  - `id, ts, request_id, client_key_id, provider_id, model_id, path, status, latency_ms, input_tokens, output_tokens, error_code`

## 8. 管理台（MVP 页面与行为）

页面（固定）：
1. **Setup Wizard**（首次启动）  
   - 设置 admin 密码  
   - 生成一个默认 client API key（可复制）  
   - 引导添加第一个 Provider（forward/claude）
2. **Dashboard**：运行状态、provider 健康、最近错误
3. **Providers**：新增/编辑/启用/禁用/健康检查
4. **Models**：模型列表、变体展示、映射编辑、从 Provider 导入
5. **Marketplace**：模块浏览、安装、启用、配置
6. **Logs**：仅元数据查询（按时间、provider、model、状态码）
7. **Docs**：内置指导手册（如何配置 Cherry Studio 等；如何使用 `model@provider`）

UI 技术（固定，避免 Node 构建依赖）：
- Go `html/template` + `embed`
- HTMX（表单与局部刷新）
- Pico.css（轻量美观，无构建）

## 9. 路线图（模块化交付，按优先级）

### v0.1（MVP）
- Core + Admin UI + SQLite
- 内置 Forward Provider + Anthropic Provider
- 模型注册表 + `model@provider` 变体
- Marketplace（GitHub 索引 + ZIP 安装 + SHA256）
- 模块运行器（HTTP OpenAI upstream + gRPC chat 最小集）

### v0.2（供应源扩展）
- 内置/模块化 Gemini Provider（官方 API：key/OAuth 任选其一，但走公开可用路径）
- Provider 能力矩阵展示（UI 上每个 provider 支持哪些 `/v1/*`）

### v0.3（高级分发模块：Distribution）
- 作为独立模块发布（仍 GPLv3 可直接合并/抽代码）
- 能力：
  - 批量生成 client keys
  - per-key 限额（按天/按月；先按“请求数/估算 token”，逐步升级为“真实 token”）
  - per-key 绑定 provider pool（专属号池/专属供应源集合）
  - 额度卡（卡密）生成/核销/给 key 充值
- UI：
  - 管理端：Key 列表、限额、池绑定、卡密生成与核销记录

### v0.4（用户端门户模块：Client UI）
- 用户用自己的 key 登录（或输入 key 验证）
- 查看用量/剩余额度/历史
- 兑换卡密充值

### v0.5（服务器运行环境模块：Server Runtime）
- TLS（自签或上传证书）
- IP allowlist/denylist
- Rate limit
- 反代配置生成（Nginx/Caddy 模板）

## 11. 关键默认值与假设（写死在实现与文档里）
- 默认监听：`127.0.0.1:3210`（UI 与 API 同端口）
- 默认开启鉴权：所有 `/v1/*` 必须 `Authorization: Bearer <client_key>`
- 模块只从官方精选索引安装；索引与模块均使用 SHA256 校验
- 日志默认仅元数据；提示词/响应正文不落盘
- API key/token 明文存储（文档必须强调：限制数据目录权限、不要暴露到公网）
- 供应源模块是否“合规/稳定”由模块自身决定；Core 只提供框架与隔离运行能力
